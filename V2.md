## 1. What this project is

A full-stack QA/SDET portfolio project. Three independently-owned test layers against a real, self-hosted OpenEMR instance (healthcare practice-management system, PHP/MySQL), standing in for the class of app a company like TherapyNotes runs in production:

- **API** — C#/xUnit, REST (`/apis/{site}/api`) and FHIR R4 (`/apis/{site}/fhir`) surfaces, `tests/OpenEmr.Api.Tests`
- **Database** — C#/xUnit, direct MariaDB connection via MySqlConnector + Dapper, `tests/OpenEmr.Db.Tests`
- **UI** — Playwright/TypeScript, page-object pattern, `ui/pages` + `ui/tests`
- **treeLine** — not a test layer; a separate AI-powered Playwright site-comprehension tool used once so far to generate a structural crawl of the running app, intended to drive what `TEST-PLAN.md` covers next

The three test layers are deliberately decoupled (no shared fixtures, no cross-project execution-order dependency) — this mirrors how real QA orgs split API/DB vs. E2E ownership, and is a repeatedly-stated non-negotiable in `CONTEXT.md` and `CLAUDE.md`.

Full narrative rationale (why OpenEMR, why C# for API/DB but TS for UI, why layers are decoupled) lives in `CONTEXT.md`'s Decision Log — not repeated here in full, only referenced where it affects prioritization below.

**C# bugs fixed in prior sessions** (per `HANDOFF.md` + commit `03d2b34 fix two api related bugs`):
1. Appointment list endpoint (`GET /patient/{pid}/appointment`) — response root is a bare array, not `{"data": [...]}`. Original test's `.GetProperty("data")` on an array-typed root caused `InvalidOperationException`. Fixed by parsing root directly as an array.
2. Double-booking — test originally expected a `409 Conflict`; confirmed via direct `curl` that OpenEMR returns `200 OK` with a new distinct `id` on both identical POSTs, no server-side conflict enforcement. Test renamed to `Double_Booking_Same_Slot_Same_Provider_Is_Allowed_With_Distinct_Ids` to assert real behavior instead of a wrong assumption.

Additional non-obvious findings baked into the Encounter/Practitioner test design (see `CONTEXT.md` Known Constraints for full detail, only summarized here):
- Encounter routes key on patient **UUID**, Appointment routes key on **pid** — not consistent across resources.
- Encounter list wraps in `data`; Appointment list does not — enveloping is inconsistent per-resource, don't extrapolate.
- Closing/signing an encounter is `PUT .../encounter/{euuid}` with `last_level_closed: "1"`, and — unlike `POST` — requires the caller to explicitly supply `user`/`group` in the body (undocumented in the OpenAPI comments).
- A Practitioner created without a `username` is permanently invisible to list/get, even though `username` isn't in the validator's required fields — root-caused to a `PractitionerService::search()` filter the OpenEMR source itself has a maintainer TODO calling "bizarre."

**Open UI bug (highest-priority UI item):** `CalendarPage.deleteCurrentEvent()` submits the delete form and accepts the native confirm dialog, but the row persists in `openemr_postcalendar_events` afterward — confirmed via direct DB check, not a timing artifact. Suspected needs the same `top.restoreSession()` call the save flow required, or the delete AJAX call needs a different completion signal. Next debugging step documented in `HANDOFF.md` (read `deleteEvent()`/`SubmitForm()` in the container's `add_edit_event.php` via `docker exec`).

**Structural discovery governing every future POM:** OpenEMR's authenticated UI is a tabbed frameset (`main.php`), not page-to-page navigation. Tabs load into iframes with stable hand-assigned names (`iframe[name="cal"]`, `iframe[name="pat"]`, etc.). `page.goto()` a second time mid-session reliably breaks the session ("Authentication Error") — every POM must switch tabs via the real top-nav menu instead. This is the single most load-bearing fact for any new UI work.

**Known, unresolved runner-specific flakiness:** legacy jQuery-dialog popups (`library/dialog.js`) are measurably less reliable under `npx playwright test` than a bare Playwright script, for reasons not root-caused despite ruling out several usual suspects. Current mitigation (invoke the dialog's JS callback directly + `retries: 2-3`) works in practice but is a documented workaround, not a fix. Affects patient-registration's duplicate-check popup and calendar dialogs.

### CI

`.github/workflows/ci.yml` exists and runs all three layers (docker stack up → API/DB tests → UI tests → artifact upload → teardown with `-v`) on `push`/`pull_request` to `main`, plus manual `workflow_dispatch`. **No scheduled (cron) trigger yet** — daily smoke / weekly full regression with failure alerting is a stated goal in `CONTEXT.md` and `HANDOFF.md`'s Immediate Next Steps but not yet built.

## 3. treeLine crawl — what it produced and its actual scope of usefulness

The user's most recent completed task (`TREELINE-BUILDOUT-SESSION.md`) ran a treeLine authenticated crawl against this project's Dockerized OpenEMR instance and used the output to build the UI layer above. Key facts for whoever picks this up next:

- **95 pages crawled**, output at `treeLine-output/openemr-qa/` (`crawl.sqlite`, `hard-pages/`, `poms/`, `reports/`, `screenshots/`, `specs/`, `skipped-elements.json`).
- **This is a UI-only artifact — confirmed, not assumed.** `treeLine-output/FEEDBACK-FOR-TREELINE.md` explicitly checked for network/XHR capture that could seed API test data and found none: `crawl.sqlite` and the JSON files only ever record page-level metadata (URL, reason code, timestamp), never request/response bodies or an endpoint list. **It does not and cannot move the needle on the C#/API/DB layers.** This is the direct answer to the user's stated framing ("hopeful this run would result in more accurate tests... probably only assist the UI suite") — confirmed correct.
- **Login itself was never crawled** — the crawl session started pre-authenticated.
- **Almost no overlap with `TEST-PLAN.md`'s open UI items** — the 95 pages are almost entirely deep admin/report/config surface.
- **Several generated POMs don't compile as-is** — concrete generator bugs, not just gaps in coverage:
  - Class names embed a literal `.php` from the source filename (`InterfaceBillingNew_payment.phpPage`) — invalid TS syntax.
  - Locator property names derived from visible combobox text can start with a digit (`readonly 800Link: Locator`) — invalid JS/TS identifier.
  - One `<select>`-heavy page produced a single property name that's the concatenation of ~150 country option names, several hundred characters long.
- **0% `data-testid` coverage across all 95 pages** — expected, since OpenEMR ships none; not a crawl defect, just dead-weight metric for this target.
- **Structurally cannot capture the frame/tab navigation model** — because the crawl loaded pages directly rather than through OpenEMR's tab-strip/iframe shell, none of the 95 atlas entries record the iframe-name/tab-navigation relationship. Every POM's `goto()` therefore needs live rewriting before it works — this cost real time on `PatientRegistrationPage` and `CalendarPage` specifically.
- **The one genuinely reusable, still-open asset:** a 95-page axe-core accessibility baseline (607 violations) at `treeLine-output/openemr-qa/reports/axe-report.md` — directly seeds the still-open "Accessibility" `TEST-PLAN.md` line item without having to build that scan from scratch.
- **One concrete finding from manually verifying a treeLine-sourced flow (not from the crawl data itself):** the real top-nav "Fees > Payment" menu does not land on the URL treeLine captured (`new_payment.php`) — it navigates to `interface/patient_file/front_payment.php`. This was found by clicking through the real UI, not by trusting the crawl, and is now documented in `CONTEXT.md` Known Constraints so it isn't rediscovered.

**Forward-looking suggestions logged for a future treeLine capability** (not to build now, just recorded): capture network requests during crawl; crawl through the app's actual navigation instead of direct URLs, recording which menu path/frame reached each page; flag client-side JS behavior (e.g. `confirm()` dialogs) that contradicts assumptions an API test might make; sanitize generated identifiers (strip dots, prefix leading digits); bound the combobox-name-concatenation heuristic.

## 6. Non-obvious constraints worth carrying into any new work (condensed from `CONTEXT.md`)

- `patient_data.pid` is application-assigned (`MAX(pid)+1`), not auto-increment — direct SQL inserts must compute it.
- Single-patient REST endpoints key on UUID, not `pid`; Appointment endpoints key on `pid`; Encounter endpoints key on `puuid` — **per-resource, don't assume one carries to the next.**
- List-endpoint response enveloping is inconsistent per resource (Appointment: bare array; Encounter: wrapped in `data`) — verify per-endpoint.
- `DEMO_MODE` doesn't work on the pinned production image tag — DB layer seeds its own fixtures instead.
- Double-booking is a soft warning at both UI and API layers, not a hard conflict — already correctly reflected in current tests.
- OpenEMR's UI is a tabbed frameset; never call `page.goto()` a second time mid-session.
- `Fees > Payment` requires a patient to already have a visit/encounter for today; the visit-creation form has an undocumented client-side validation rule (`pc_catid` must not stay on its placeholder `_blank` value) that fails silently with no visible on-page error.

Full detail and rationale for each of these lives in `CONTEXT.md`'s Known Constraints and Decision Log sections — this list is a pointer, not a replacement.
